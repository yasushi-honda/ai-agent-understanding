アーキテクチャ図解

LLM → MCP → FC → Function のシンプルな流れ AIエージェントの実行プロセスをブロック単位で簡潔に説明
MCPが支えるFunction Callingアーキテクチャ エージェントホスト、MCPサーバー群、外部サービスの関係
Function Calling と MCP の関係性 MCPはFunction Callingを簡単に使えるようにした規格統一パッケージ
非エンジニア向け: AIが自律的に外部と連携する仕組み メール、チャット、データベース、カレンダーなど外部サービスとの接続
AIコーディングエージェントによるMCPサーバー自動生成フロー システム情報読み込み → アーキテクチャ分析 → MCP定義変換 → 完成

概念理解

01. LLMの基礎

大規模言語モデル = AIエージェントの「脳」

  • ユーザーの意図を汲み取る
  • どのツールを呼ぶか判断
  • 結果を解釈して次のアクションを決定

02. Function Calling

LLMが外部機能を呼び出す万能インターフェース

  • 関数名 + 引数をJSON形式で生成
  • ホスト側が実際の関数を実行
  • 実行結果をLLMに返却

03. MCP

Function Callingの規格統一パッケージ

  • Tools / Resources / Prompts の3要素
  • Plug&Playで疎結合に拡張
  • OAuth 2.1 / Streamable HTTP

04. エージェントパターン

ReAct, Plan&Execute, Reflection等

  • シンプルなパターンから始める
  • 必要に応じて複雑さを増す
  • Generator-Evaluator構造

05. マルチエージェント

複数エージェントの協調アーキテクチャ

  • Orchestrator-Workers
  • Peer-to-Peer / Pipeline
  • Agent Teams（Claude Code）

ベストプラクティス

Anthropicの設計哲学

1

最もシンプルな解決策を最初に

必要な場合にのみ複雑さを増す。6つのComposableパターンを下から順に検討。

2

ハーネスが結果を決める

モデルの能力を引き出すのはハーネス（外枠）設計。Generator-Evaluator構造で品質を担保。

3

コンテキスト管理が最重要

エージェント開発で最も困難な問題。70-80%で品質劣化開始、最後の20%で80%のエラー発生。

MCPサーバー設計

4

1サーバー = 1責務

小さく焦点を絞ったサーバーを作る。動的ツールローディングでトークン消費を最適化。

5

最小権限 + OAuth 2.1

エージェントに必要な権限だけ付与。2026年の認証標準はOAuth 2.1。

6

Streamable HTTP + Blue-Green Deploy

本番はStreamable HTTPトランスポート。ゼロダウンタイム更新のためBlue-Greenデプロイ。

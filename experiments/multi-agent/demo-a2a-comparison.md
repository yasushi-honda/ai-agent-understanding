# A2A vs MCP協調の比較メモ

*調査日: 2026-04-05*

## 実験結果

### 1. Agent Card方式 vs MCP tool listing

| 観点 | A2A Agent Card | MCP tool listing |
|------|---------------|-----------------|
| **能力の公開方法** | `/.well-known/agent.json` で静的に公開 | `tools/list` RPC で動的に取得 |
| **記述の粒度** | エージェント単位（高レベル） | ツール単位（低レベル） |
| **発見メカニズム** | DNS/HTTPベースの自動発見 | 設定ファイルでの明示的登録 |
| **認証情報** | Agent Cardに含む | 接続時のトランスポート設定 |

### 2. 通信プロトコルの違い

| 観点 | A2A (gRPC) | MCP (JSON-RPC 2.0) |
|------|-----------|-------------------|
| **オーバーヘッド** | バイナリ、低オーバーヘッド | テキスト、人間可読 |
| **ストリーミング** | 双方向ストリーミング | SSE（サーバー→クライアント） |
| **デバッグ容易性** | 低い（バイナリ） | 高い（JSON） |
| **ツールサポート** | Protocol Buffers必要 | JSON Schemaで十分 |

### 3. 実感した利点/課題

**MCP協調の利点:**
- 既存のMCPインフラをそのまま活用できる
- ツール単位の粒度でオーケストレーションが自然
- Claude CodeなどのAIホストとの統合が容易

**MCP協調の課題:**
- エージェント間の「意図」の伝達が困難（ツールレベルの情報しかない）
- 動的なエージェント発見メカニズムがない
- 長時間タスクの進捗通知が標準化されていない

**A2Aの期待される利点（未検証）:**
- エージェント単位の能力記述で意図が明確
- タスク状態のライフサイクル管理が標準化
- Streaming対応で長時間タスクの進捗通知が可能

## 結論

現時点（2026年4月）では、MCPベースの協調が実用的。
A2Aは「エージェント間の意図伝達」と「動的発見」という
MCPが弱い領域を補完する可能性が高い。

**推奨スタック:**
```
アプリケーション層:  A2A（エージェント発見 + タスク委譲）
ツール接続層:        MCP（ツール呼び出し + データアクセス）
```

両方が成熟した段階で、coordinatorをA2Aベースに移行することを検討。
